一、 設計背景與核心痛點 (Background & Problem Statement)
在中小型倉庫管理與外勤業務報價中，傳統「商品條碼識別」面臨以下嚴重瓶頸：

1. 【手動輸入品項極易出錯且耗時】：
   倉庫人員在點貨或出貨時，必須在小螢幕手機上人眼識別條碼貼紙，然後手動敲入 13 位的商品條碼。這常因視覺疲勞敲錯數字，導致庫存系統與帳務錯亂。

2. 【客製化手持盤點槍成本昂貴】：
   購買專用的工業級手持盤點槍（PDA），每台動輒數萬元台幣，且需要跟舊 ERP 系統進行繁瑣的軟體開發對接，對於小規模企業是巨大的預算負擔。

3. 【APP 安裝阻力與跨平台痛點】：
   如果專門開發一個手機 App，面臨 Android 與 iOS 系統相容性問題，且一線員工對於「安裝新軟體」通常有較高的抗拒感，不易導入推行。

★ 系統設計核心原則：「零安裝、對準即掃、即時反饋」。利用 Telegram Mini App（小程序）技術，直接在對話框內拉出相機鏡頭，員工不需安裝任何 App，直接用自己的手機就能秒級掃碼查詢 ERP 庫存。

二、 系統模組與執行流程 (System Modules & Process Flow)
本系統由以下模組無縫對接運作：

1. 【本機設定伺服器 (take_server.js)】：
   提供 HTTPS 網頁靜態路由，將掃碼網頁伺服至公網，並接收來自前端網頁的條碼查詢請求。

2. 【內嵌相機掃碼網頁 (scanner.html)】：
   使用 HTML5 + JS 撰寫，整合開源高性能解碼庫，並引入 Telegram WebApp SDK。

3. 【Telegram 橋接器 (telegram_bridge.js)】：
   在 Bot 選單中註冊 Web App 按鈕，當收到網頁傳回的條碼時，自動在背景讀取 VFP 商品庫存檔並將規格、庫存、售價格式化傳給使用者。

執行流程簡圖：
    [業務手機 Telegram] ── 點擊「📷 開啟相機」 ──► [彈出 Mini App Web 視窗]
                                                        │
                                                        ▼ (請求並啟用相機)
    [Bot 接收到條碼] ◄── 傳送條碼資料 (自動關閉) ◄── [對準條碼, 自動辨識成功]
           │
           ▼ (查詢本機 ougo.dbf)
    [手機收到品名/庫存/售價報表]

三、 核心技術發想與運作機制 (Core Technical Details & Write Mechanism)

1. 🚀 Telegram Mini App 內嵌網頁秒級調用：
   - 傳統網頁需要使用者開啟瀏覽器、輸入網址、登入等步驟。
   - 本系統利用 Telegram 官方的 WebApp 協定，當使用者點擊 Bot 按鈕時，會直接在 Telegram 介面內向上滑出一個 WebApp 視窗，提供如同原生 App 的流暢體驗。

2. 🔍 HTML5 實時串流與 html5-qrcode 高速解碼：
   - 網頁前端使用 `html5-qrcode` 套件，透過瀏覽器的 `navigator.mediaDevices.getUserMedia` 直接取得手機後置主鏡頭的影像串流。
   - 畫面中心會顯示一個「掃描定位框」，背景以每秒 15 幀的速度進行影像二進位解碼，支援 EAN-13, Code 128 等主流一維條碼及 QR Code，對準後在 100-200 毫秒內即可完成識別。

3. 📡 輕量化 JSON API 查詢與資料無縫整合：
   - 當解碼成功後，網頁會觸發手機震動回饋，並立即透過 `Telegram.WebApp.sendData(barcode)` 將結果傳回給 Telegram Bot，同時自動關閉網頁。
   - Bridge 接收到該事件後，會將條碼字串傳給唯讀引擎查詢庫存，讓使用者在對話框內秒收結果，達到最流暢的「一鍵掃碼 -> 獲取結果」閉環。

四、 安全防護設計 (Security Architecture)
1. 【HTTPS 傳輸加密與相機限制】：手機系統強制規定必須在 HTTPS 安全加密協議下才允許調用相機權限。我們的 WebApp 會透過 Localtunnel 所分配的安全 HTTPS 網址進行發布，確保影音串流不被竊聽。
2. 【API Token 來源驗證】：WebApp 網址中會帶有一次性加密 Token，只有從授權的 Telegram 對話框點擊進來的請求，`take_server.js` 才會提供掃碼頁面，防止外部惡意掃描。

五、 系統討論重點 (Key Discussion Points)

1. 【相機焦距與低光源適應度】：
   * **問題**：倉庫深處常有光線不足問題，或手機鏡頭無法自動對焦條碼。
   * **優選方案**：
     - 在掃描畫面上方增加一個「開啟閃光燈（手電筒）」的輔助按鈕（部分支援 WebRTC 的手機瀏覽器可調用）。
     - 設定解碼框大小佔比，引導使用者將條碼移至中央以利自動對焦。

2. **條碼重疊或多條碼干擾**：
   * **問題**：若包裝盒上同時貼有原廠條碼與貨架標籤，相機可能會誤掃。
   * **建議**：解碼成功後先顯示掃到的條碼數值並暫停，由使用者點選確認，或在畫面中提供更窄的掃描視窗以利精準對位。
