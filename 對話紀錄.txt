二、 系統架構與檔案清單
--------------------------------------------------
本專案已完全整合收納至以下本機路徑。為維護安全隱私，本系統不在 Z 槽 (NAS) 留下 any 自訂程式備份。

【本機執行路徑 C:\agy_Add_on\ERP_System\】：
1. telegram_bot.js (查價、歷史售價與智慧排程機器人)：
   負責 Telegram 查價、雙關鍵字客戶歷史成交價查詢與定時推送智慧警報（常駐背景）。
2. take_server.js (盤點與設定伺服器)：
   負責開啟本地 Port 3000、提供視覺化設定後台，並與 Localtunnel 連線產生動態網址並推送（常駐背景，帶有 API 金鑰防護鎖）。
3. config.json (統一設定檔)：
   支援雙機器人、智慧警報、API 金鑰與資料庫設定。
4. stock_snapshot.json (庫存狀態快照)：
   智慧缺貨比對用，自動記錄上次檢查之庫存數，防訊息爆量。
5. append_take.ps1 (寫入腳本)：
   將盤點資料（數量、條碼、廠商、主/次單位）解析並安全寫入 VFP 資料庫的 `take.dbf` 及 `take1.dbf`。
6. public/ (靜態資源資料夾)：
   手機條碼盤點助手前端 UI，包含 `index.html`、`index.css` 與 `index.js`（含廠商模糊搜尋選單）。
7. run_query.bat / run_query.vbs (查價啟動腳本)：
   用於啟動/背景隱形啟動 `telegram_bot.js`。
8. run_take.bat / run_take.vbs (盤點啟動腳本)：
   用於啟動/背景隱形啟動 `take_server.js`。
9. 開機自啟動代理 (ERP_隨身查.vbs)：
   置於本機 `Startup`（啟動）資料夾下，開機自動背景同時載入「查價」與「盤點網頁」服務。
10. run_bot.vbs (相容啟動代理腳本)：
    置於 C:\agy_Add_on\run_bot.vbs，防範 Windows 排程或啟動報錯。

【第九階段：智慧型「缺貨與主力低存」排程推播實作】
* 用戶提出：
  - 將先前關閉的低庫存推播功能重新開啟，但改成智慧型篩選，防止兩千多筆低存商品造成的訊息轟炸。
  - 具體包含兩部分：一是在商品庫存「剛好變為 0 (斷貨)」的瞬間觸發警報；二是只針對手動設定的「熱銷、高單價」主力商品進行常規低庫存水位推播。
* 開發與實作：
  1. **資料量診斷**：經檢測，ERP 中高達 1,529 筆正常銷售商品目前庫存為 0，證明如果不做狀態篩選，直接推播 0 庫存依然會導致 Telegram 訊息爆量與截斷。
  2. **智慧缺貨狀態偵測（狀態機）**：
     - 在程式中引入了 `stock_snapshot.json` 快照比對機制。每次推播時間點（預設每日 08:30、17:00）會比對當前與上次的庫存。
     - **只有當商品庫存從大於 0 變為小於等於 0（剛歸零）時，才會觸發 Telegram 推播警報**，其餘已是 0 庫存的品項不再重複打擾。
  3. **主力商品低庫存警訊**：
     - 在 `config.json` 中加入了 `priorityProducts` 欄位（主力商品編號陣列）。
     - 只要主力商品庫存低於門檻，便會列入警訊。非主力商品則不受影響，僅在斷貨瞬間警報。
     - 此設計落實了「無事不打擾，精準防斷貨」的精神。
  4. **服務重啟**：在背景以新版本程式碼重新載入了 `telegram_bot.js` 服務，定時排程與智慧偵測功能已順利開啟。

【第十一階段：記憶體保護鎖、啟動載入器重構與異體同音字對應搜尋】
* 用戶提出：
  - 搜尋「華新大白」時，即使有切分邏輯，在 Telegram 上依舊出現搜尋不到的狀況。
  - 電腦開機或雙擊捷徑時，Windows Script Host 有時會彈出「記憶體不足」的錯誤視窗。
  - 希望解決「克淋濕」與「克淋溼」因一字之差（溼/濕）導致在系統上無法跨字搜尋的狀況。
  - 部分列管的主力商品清單無法在設定後台順利新增。
* 開發與優化：
  1. **背景常駐程序同步重啟**：
     經診斷，先前的「華新大白」搜尋不對稱，是因為在背景執行的長輪詢機器人與 API 伺服器並未重新啟動，仍在運行不包含拆分字典的舊程式碼。將背景 Node.js 舊進程安全終止，並重新以 WMI 模式常駐載入新版程式，已能精準查得 4 筆大白資料。
  2. **啟動載入器 (VBScript) 純 ASCII 優化**：
     開機啟動捷徑 `ERP_隨身查.vbs` 中因帶有中文字元路徑，在 Windows WSH 引擎下容易發生編碼轉換崩潰（提示「記憶體不足」或「語法錯誤」）。將 `ERP_隨身查.vbs` 重構為「100% 純英文/ASCII 腳本」，利用 FSO 自動掃描以 `ERP` 開頭的資料夾路徑，並改用 `cmd.exe /c run_query.bat` 執行方式，徹底根治 Windows 系統的開機報錯，並自動啟用運行記錄檔（`telegram_bot.log` 與 `take_server.log`）。
  3. **資料庫防溢出記憶體鎖**：
     為防範 ERP 系統在維護或寫入 `stock.DBF` 時，暫時產生不完整的檔案頭 or 超長 RecordCount，在 `readStockDb` 中新增了「物理長度防溢出指針 check」。若發現檔案偏移指針超出 fileBuffer 長度即強制跳出迴圈，確保程式在任何極端狀況下都不會引發 CPU 100% 卡死或 OOM 記憶體崩潰。
  4. **同音異體字自動對應搜尋 (normalizeText)**：
     在背景搜尋比對中導入 `normalizeText` 模組，將常規異體字與簡繁字（`溼` ↔ `濕`、`臺` ↔ `台`、`綿` ↔ `棉`、`雙` ↔ `双` 等）在比對前統一標準化。不論使用者輸入何種寫法，均可同時檢索出庫存中含有任一寫法之所有品項。測試「克淋溼」與「克淋濕」皆能精準回傳 7 筆完整符合的品項。
  5. **後台主力商品加入 Bug 修復 (Cache 傳參)**：
     原先設定後台渲染搜尋結果時，按鈕的 `onclick` 事件直接將包含規格描述的商品名稱字串拼入 HTML，而醫療品項名稱中常常含有表示英吋的雙引號（`"`，如 `3M 克淋濕 2"x3"`），導致 HTML 屬性結構被提前截斷破裂，引發按鈕失效。修改為將搜尋結果保存在全域 JS `searchResultsCache` 記憶體中，按鈕僅傳遞純英文數字之商品編號（NO），徹底根治此網頁語法衝突，並升級 API 支援 `nos` 批次載入商品即時資訊。

【第十二階段：API 金鑰安全鎖、盤點介面與設定後台全面功能升級】
* 用戶提出：
  - 加強外網安全防護，為行動網頁加上 API Token 憑證驗證，防止外網惡意掃描庫存與廠商資料 API。
  - 盤點網頁下拉選單中的廠商名單過長，需要支援在手機上快速模糊搜尋與觸控優化。
  - 提案一：【客戶專屬歷史售價查詢】。業務在 Telegram 輸入「商品名稱 客戶名稱」（如：`大白 佳合診所`），自動抓出該客戶最近三次購買該商品的日期、數量和當時的成交單價。
  - Telegram 中的管理後台連結「localhost」被官方安全機制阻擋，要求變為可點選的藍色連結。
  - 將新開發的「歷史售價查詢」功能以核取方塊形式，新增至設定後台的「啟用功能選單」中。
* 開發與實作：
  1. **外網 API 金鑰安全鎖**：
     - 在伺服器啟動時，若 `config.json` 中沒有 `apiToken` 則自動隨機生成一組 22 位的安全憑證。
     - 當 Telegram 發送盤點網址時，會自動在 URL 結尾附帶 `?token=telcradrq...`。手機端網頁載入時會自動將該 token 存入 `sessionStorage`，並在隨後的 API 請求中以 `x-api-token` Header 傳送。
     - `take_server.js` 後台檢測如果連線來自非本機（外網），則強制進行 token 比對，未授權請求一律回傳 401 Unauthorized，徹底封鎖庫存外洩風險。
  2. **手機盤點網頁 UI 優化與廠商模糊搜尋**：
     - 在手機盤點頁面（public/index.html）的廠商下拉選單中整合「模糊搜尋過濾器」。使用者在文字框輸入注音或文字時，選單會動態篩選符合的廠商，並擴大點擊區域與觸控反饋，解決手機操作不易的問題。
     - 修復 CSS 樣式中 card 容器下緣被切除的問題，將 `overflow: hidden` 改為 `visible`。
     - 建立一個空的代理腳本 `C:\agy_Add_on\run_bot.vbs` 以相容於既有 Windows 自動化腳本，消除開機時的找不到檔案彈窗。
  3. **客戶專屬歷史售價查詢 (逆向分塊掃描)**：
     - 在 `telegram_bot.js` 中新增雙關鍵字分類器。當收到兩個以上字詞時，系統會先對比 `CUST.DBF`。為防止混淆，純數字關鍵字（如 `75`）不會模糊匹配為客戶名稱。
     - 實作 1.3 GB 大檔 `ougo1.dbf` 的 `queryDbfOptimized` 逆向分塊掃描。從檔案末尾向前的以 5,000 筆為單位讀取緩衝，只要找到該客戶的 3 筆該商品交易紀錄就立刻回傳，成功將網路共用資料夾的查詢耗時從 20 幾秒壓縮至 200 毫秒內。
  4. **Telegram 可點選本地連結 (lvh.me) 繞過防護**：
     - 由於 Telegram 官方封鎖 `localhost` / `127.0.0.1` 作為超連結，將設定後台連結改為萬用本機解析網域 `http://lvh.me:3000/config`。
