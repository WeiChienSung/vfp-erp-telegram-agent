一、 設計背景與核心痛點 (Background & Problem Statement)
在傳統醫材與化工批發 ERP 營運中，熱銷商品突然斷貨常導致出貨中斷，造成客戶流失與業績損失。現有的庫存通報機制有以下痛點：

1. 【告警時間滯後】：
   傳統通知多採「每日下班報表」或「每日定時推播」。若某商品在早上 09:00 即賣斷貨，採購人員往往要等到下午或傍晚才得知，錯失了當天向供應商追貨或緊急調貨的黃金時間。

2. 【警報疲勞 (Alert Fatigue)】：
   大多數庫存通報系統在商品庫存為 0 時，會每天重複發送通知。對於長期斷貨或已停售商品，人員每天接收數百條無效通知，最後習慣性忽視警報，導致「剛剛才賣斷貨」的真正警報被淹沒。

3. 【行動端回查操作繁瑣】：
   採購與業務收到斷貨推播後，最迫切的動作往往是回查該商品「最近賣給誰、售價多少、有沒有未出貨單據」以進行調度。然而，在手機上手動輸入關鍵字查歷史價或欠單操作極度繁瑣，影響反應速度。

★ 系統設計核心原則：「即時響應、精準降噪、一鍵互動」。每分鐘自動掃描，只在「有貨變無貨」瞬間發出警報，並提供 Inline 按鈕一鍵回查歷史成交。

二、 系統模組與執行流程 (System Modules & Process Flow)
本系統主要由兩個常駐背景模組協同運作：

1. 【即時售罄過濾器 (zero_stock_alert.js)】：
   以輕量定時輪詢（每 60 秒）監控 VFP 的商品庫存檔 (ougo.dbf)，僅在狀態變更的臨界點觸發推播。

2. 【Telegram 行動互動模組 (Callback Query Handler)】：
   在推送訊息中動態嵌入 Inline Keyboard（互動式按鈕），接收使用者的按鈕點擊，並在背景直接執行歷史交易撈取。

運作生命週期流程圖：
   [VFP ougo.dbf] 
        │ (每60秒掃描)
        ▼
   [狀態機檢測: prevStock > 0 && currentStock <= 0]
        │ 
        ├─► (否) ──► [過濾忽略, 保持靜音]
        ▼ (是)
   [售罄打包 (Debounce)] ──► [發送 Telegram 推播 (帶有 Inline 查詢按鈕)]
                                  │
                                  ▼ (業務在手機點選「一鍵歷史成交」)
                           [呼叫 answerCallbackQuery 消除轉圈] ──► [快閃呈現歷史成交紀錄]

三、 核心技術發想與演算法設計 (Core Technical Details & Algorithms)

1. 🔄 庫存狀態機過濾與冷啟動保護 (State-Machine Filtering & Cold-Start Guard)
   * 發想由來：
     為了解決重複警報洗版與服務重啟時的誤報問題。
   * 設計實現：
     - 系統在記憶體中維護當前的庫存狀態。僅當某商品庫存「從上次大於 0，在本次掃描變為小於或等於 0」時才判定為售罄。
     - 狀態轉換表：
       ┌───────────────┬───────────────┬────────────────┐
       │ 上次庫存狀態  │ 當前庫存狀態  │ 系統動作與推播 │
       ├───────────────┼───────────────┼────────────────┤
       │     > 0       │     > 0       │ 靜音，不推播    │
       │     > 0       │    <= 0       │ 觸發【售罄告警】│
       │    <= 0       │    <= 0       │ 靜音，避免疲勞  │
       │    <= 0       │     > 0       │ 重置狀態為有貨  │
       └───────────────┴───────────────┴────────────────┘
     - 【冷啟動防護 (Cold-Start Guard)】：當背景服務重新啟動時，會先從硬碟載入最後的庫存快照 `stock_snapshot.json`；若快照不存在，則以當前最新庫存建立初始快照，且首輪掃描不發送任何警報，避免重啟服務時發送大量歷史已斷貨商品的垃圾訊息。

2. 📦 多重事件防抖動合併打包 (Debounced Alert Batching)
   * 發想由來：
     如果同一分鐘內有多個商品同時賣完，機器人如果分開連續發送多則訊息，除了會造成使用者手機頻繁震動打擾，還會觸發 Telegram API 的流量限制 (Rate Limit)。
   * 設計實現：
     - 在每分鐘的輪詢比對中，系統會將所有在該分鐘內「新賣完」的商品暫存在一個陣列中。
     - 完成該輪掃描後，將所有售罄商品打包成一則結構化訊息一次發送。

3. 🖱️ Telegram Inline Keyboard 一鍵歷史成交背景查詢 (Callback Query Handler)
   * 發想由來：
     點擊訊息中的「一鍵歷史成交」按鈕，能讓使用者在手機上省去輸入關鍵字查歷史價的繁瑣操作。
   * 設計實現：
     - 系統在告警訊息下方動態生成帶有商品代號（MNO）的 Inline Button（回呼資料為 `hist:MNO`）。
     - 當使用者點擊時，Telegram 會發送一個 `callback_query` 到 Node.js 背景。背景程式接到後，立即調用「1.3GB 逆向分塊檢索演算法」，在 200ms 內於背景查出最近的成交明細，並將結果回傳。
     - 系統同時調用 `answerCallbackQuery` 清除 Telegram 介面上的轉圈載入狀態，提供極致流暢的行動版使用者體驗。

四、 安全防護與效能優化 (Security & Performance)
1. 【背景執行緒防堵塞】：Callback Query 的歷史成交查詢可能在幾秒內被多個業務同時點擊。由於 Node.js 是單執行緒，逆向分塊讀取必須維持高效率。透過 Lazy Decoding 優化，確保單次歷史撈取不阻塞主執行緒。
2. 【API 安全限流保護】：推播時主動進行批次打包，並在背景對單一 Chat ID 進行發送頻率限制，符合 Telegram 每秒最多 30 則訊息的安全合規限制。

五、 系統討論重點 (Key Discussion Points)

1. 【營業時間外的輪詢策略】：
   * **問題**：目前晚上 21:00 下班後至隔天 08:00 雖然 ERP 不會修改檔案，程式會使用快取，但仍有輪詢開銷。
   * **分析與建議**：
     - 建議在非營業時間（晚上 9 點到早上 8 點）直接暫停 Polling 定時器。這不僅能將 CPU 磁碟 I/O 降為絕對的零，也能防止深夜時段因特殊系統維護導致的任何誤報打擾。

2. 【告警群組分流訂閱】：
   * **問題**：目前所有售罄警報是發送到同一個 Telegram 管道。隨商品品項增多，非該品項負責的採購會收到多餘的訊息。
   * **分析與建議**：
     - 未來可根據 ERP 商品資料庫中的「大類程式碼」或「主要供應商」，在設定後台建立分流規則（例如：耗材類警報發至 A 群組；儀器類發至 B 群組），提升資訊的精準度。

3. 【冷啟動狀態補發機制】：
   * **問題**：若伺服器在白天營業時間中斷重啟，這期間剛好售罄的商品，因冷啟動保護會被默默忽略，導致漏報。
   * **分析與建議**：
     - 可以建立「與實體 stock_snapshot.json 快照比對」的機制。重啟時先讀取舊快照，若比對發現有商品在重啟期間歸零，則進行補發。但若斷電時間長（如跨日），則維持靜音以防訊息過時。
