========================================================================
   【VFP ERP 即時售罄告警與互動查詢系統 - 深度發想與系統架構設計書】
========================================================================

一、 設計背景與核心痛點 (Background & Problem Statement)
在傳統 ERP 管理中，採購與業務人員常面臨熱銷商品斷貨造成的業績損失。現有的庫存報表或每日通知有以下痛點：
1. 【告警時間滯後】：一般的低庫存通知採「每日定時推播」或下班報表，當商品在營業時間剛開盤即賣斷貨時，採購人員無法在第一時間得知，錯失補貨黃金時間。
2. 【警報疲勞 (Alert Fatigue)】：大多數系統在商品庫存為 0 時會每天重複發送通知，造成人員習慣性忽視警報，讓真正需要處理的「剛賣完」警報被淹沒。
3. 【行動端回查繁瑣】：當人員收到斷貨警報後，下一步通常需要立即查詢該商品「最近的成交歷史與價格」以做補貨評估，但在手機上手動輸入關鍵字搜尋極不方便。

二、 系統模組與執行流程 (System Modules & Process Flow)
本系統主要由兩個常駐背景模組協同運作：
1. 【即時售罄過濾器 (zero_stock_alert.js)】：以輕量定時輪詢（每 60 秒）監控 VFP 的商品庫存檔 (ougo.dbf)，僅在狀態變更的臨界點觸發推播。
2. 【Telegram 行動互動模組 (Callback Query Handler)】：在推送訊息中動態嵌入 Inline Keyboard（互動式按鈕），接收使用者的按鈕點擊，並在背景直接執行歷史交易撈取。

三、 核心技術發想與演算法設計 (Core Technical Details & Algorithms)

1. 🔄 庫存狀態機過濾與冷啟動保護 (State-Machine Filtering & Cold-Start Guard)
   * 發想由來：為了解決重複警報洗版與服務重啟時的誤報問題。
   * 設計實現：
     - 系統在記憶體中維護當前的庫存狀態。僅當某商品庫存「從上次大於 0，在本次掃描變為小於或等於 0」時才判定為售罄。
     - 【冷啟動防護 (Cold-Start Guard)】：當背景服務重新啟動時，會先從硬碟載入最後的庫存快照 `stock_snapshot.json`；若快照不存在，則以當前最新庫存建立初始快照，且首輪掃描不發送任何警報，避免重啟服務時發送大量歷史已斷貨商品的垃圾訊息。

2. 📦 多重事件防抖動合併打包 (Debounced Alert Batching)
   * 發想由來：如果同一分鐘內有多個商品同時賣完，機器人如果分開連續發送多則訊息，除了會造成使用者手機頻繁震動打擾，還會觸發 Telegram API 的流量限制 (Rate Limit)。
   * 設計實現：
     - 在每分鐘的輪詢比對中，系統會將所有在該分鐘內「新賣完」的商品暫存在一個陣列中。
     - 完成該輪掃描後，將所有售罄商品打包成一則結構化訊息一次發送。

3. 🖱️ Telegram Inline Keyboard 一鍵歷史成交背景查詢 (Callback Query Handler)
   * 發想由來：點擊訊息中的「一鍵歷史成交」按鈕，能讓使用者在手機上省去輸入關鍵字查歷史價的繁瑣操作。
   * 設計實現：
     - 系統在告警訊息下方動態生成帶有商品代號（MNO）的 Inline Button（回呼資料為 `hist:MNO`）。
     - 當使用者點擊時，Telegram 會發送一個 `callback_query` 到 Node.js 背景。背景程式接到後，立即調用「1.3GB 逆向分塊檢索演算法」，在 200ms 內於背景查出最近的成交明細，並將結果回傳。
     - 系統同時調用 `answerCallbackQuery` 清除 Telegram 介面上的轉圈載入狀態，提供極致流暢的行動版使用者體驗。

四、 安全防護與效能優化 (Security & Performance)
1. 【背景執行緒防堵塞】：Callback Query 的歷史成交查詢可能在幾秒內被多個業務同時點擊。由於 Node.js 是單執行緒，逆向分塊讀取必須維持高效率。透過 Lazy Decoding 優化，確保單次歷史撈取不阻塞主執行緒。
2. 【API 安全限流保護】：推播時主動進行批次打包，並在背景對單一 Chat ID 進行發送頻率限制，符合 Telegram 每秒最多 30 則訊息的安全合規限制。

五、 系統討論重點 (Discussion Points)
1. 【營業時間外的輪詢策略】：目前晚上 21:00 下班後至隔天 08:00 雖然 ERP 不會修改檔案，程式會使用快取，但為了徹底降低硬碟 I/O，是否應直接在非營業時間關閉輪詢定時器？
2. 【告警群組分流訂閱】：目前所有售罄警報是發送到同一個 Telegram 管道。隨著商品品項增多，未來是否需要支援根據「商品大類」或「供應商」將告警群組分流發送（例如：A 類耗材發給採購組 A，B 類儀器發給採購組 B）？
3. 【冷啟動狀態補發機制】：若伺服器在白天因故斷電 10 分鐘，這段期間內售罄的商品，在系統重啟後會被冷啟動保護默默忽略。是否需要建立「比對快照並補發中斷期間斷貨警報」的機制，或者維持靜音以防打擾？
