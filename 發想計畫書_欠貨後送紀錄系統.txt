   【ERP 待補貨與欠貨後送紀錄管理系統 - 深度發想與系統架構設計書】
========================================================================

一、 設計背景與業務痛點 (Background & Problem Statement)
在傳統醫材與化工原料批發業務中，常面臨「欠貨（後送）」的管理難題：
1. 【欠貨紀錄零散】：出貨時若庫存不足，開單人員會直接在銷貨單備註中手寫註記「後送 10+2」或「欠 5 盒」。這些資料散落在數萬張單據中。
2. 【手動核銷耗時】：內勤人員每天需以人工翻閱出貨單，抄錄欠貨資料到 Excel。當後續正常出貨時，又需人工逐筆去對帳、劃銷，極易發生漏單、超量重複送貨，或客戶抱怨舊貨未補。
3. 【協同覆寫衝突】：多人同時操作與編輯 Google 試算表時，容易因儲存格位移導致資料列錯位，甚至在背景同步覆寫時，將內勤辛苦手寫的「內勤手動備註」全部清空抹除。
4. 【排他鎖競態衝突】：當本機同步服務與網頁端使用者同時存取試算表時，Google Sheets 的非同步讀寫會引發嚴重的競態問題（Race Conditions），導致行號移位或出貨狀態錯亂。

本系統發想並設計了「本機 VFP 逆向檢索去重 ➔ Web Webhook 自動對齊 ➔ 試算表排他鎖與複合鍵防覆寫」的全自動資料同步閉環。

二、 核心功能組成與運作流 (Core Features & Workflow)

1. 🔍 本機 VFP 資料庫定期逆向掃描 (`sync_backorders.js`)
   * 正規表達式匹配：自動掃描銷貨明細，使用 regex 精準捕獲備註欄中的欠貨關鍵字與數量（例如將「後送12+2」自動解析拆分為 12 瓶付費品與 2 瓶搭贈品）。
   * 排除結案特例：如銷貨數量為負數（退貨）或備註包含「已取回」、「請取回」等，代表該欠貨已結案，系統自動排除。

2. 🔄 智慧去重與出貨累計扣減演算法 (Smart Shipment Deduction)
   * 演算法邏輯：
     - 同一客戶 (`customerNo`) 與同一品項 (`productNo`) 的組合可能在不同日期發生多次銷貨。
     - 系統在逆向掃描（從新到舊）時，會為每個組合鍵建立一個「已出貨數量累計器 (`shippedQtyMap`)」。
     - 當遇到「無後送備註的正常出貨記錄」時，系統會累加其出貨數量。
     - 當遇到「有後送備註的欠貨記錄」時，系統會先以該組合鍵當前累積的出貨量進行抵扣。
     - 若出貨量抵扣完欠貨量，則代表該欠貨在後續的單據中已完成出貨，該欠貨單自動「結案/去重」而不予顯示；若未抵扣完，則計算剩餘欠貨數量（新付費量與贈品量）並列為「活躍欠貨」。這實現了「出貨後欠貨自動消失」的去重防護。

3. 📝 內勤手動備註複合鍵對齊防護演算法 (Composite Key Alignment)
   * 解決同步覆蓋手動備註的痛點：
     - 在每次同步寫入試算表前，Apps Script 會先讀取主表，將「現有行」的手動備註建立一個記憶體快取。
     - 快取的 Key 採用精確的「三聯複合鍵」：`客戶代號_產品品號_出貨單號`。這能精確區分同客戶同品項但在不同出貨單發生的多筆獨立欠貨。
     - 若有些歷史手動列無單號，則系統會自動啟用 Fallback 機制，降級退化成 `客戶代號_產品品號` 來完成對齊。
     - 同步覆寫時，系統會依複合鍵將舊有的「內勤手動備註」精準填回對應的新行中，確保手寫資訊永久保留。

4. 🧹 批次出貨結案搬移演算法 (Batch Archiving)
   * 解決 onEdit 刪除行位移 Bug：
     - 傳統做法是當使用者一勾選 F 欄「確認出貨」的 Checkbox 時，Apps Script 立即調用 `deleteRow(i)` 刪除該行並複製到備份區。但在多人併發勾選時，先刪除的行會導致下方的行號向上位移，使得後續勾選的行號發生偏位，誤刪錯資料。
     - 本系統發想了「整表掃描、逆向刪除」演算法：當觸發 `onEdit` 時，不立刻刪除。而是先掃描全表，收集所有被勾選的列數據，一次性附加寫入「備份區」並加上完成時間；然後對勾選的行號進行降序排序，由下往上（逆向）刪除。這確保了每次刪除都不會影響上方待刪除行的行號，徹底解決位移 Bug。

5. 🔒 Apps Script 排他鎖機制 (Script Lock)
   - 在 `syncBackorders` (本機同步寫入)、`archiveCheckedRows` (內勤手動結案)、`onEdit` (即時打勾搬移) 三大雲端函數首行，均調用 `LockService.getDocumentLock()`。
   - 分別設置 30秒/20秒/15秒 的等待鎖。任何一方正在寫入或搬移資料時，其他併發請求必須排隊等待，直到鎖定釋放，從根本上防範並發衝突。

6. 📐 格式相容自動左移遷移器 (Layout Migrator)
   - 由於系統升級將舊版 13 欄結構（含 Column E: 出貨數量）升級為 12 欄精簡結構。
   - 雲端函數首行會調用 `checkAndMigrateLayout`。一旦檢測到 Column E 的標題為 "出貨數量"，會自動刪除第五欄，並將右側欄位（包括備註、客戶代號、品號、單號）無損左移對齊並刷新快取，防範因欄位數變動造成的資料毀損。
