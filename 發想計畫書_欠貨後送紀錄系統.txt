一、 設計背景與核心痛點 (Background & Problem Statement)
在中小型批發企業的營運中，因缺貨或庫存不足產生的「欠貨（未出貨單據）」管理是一大痛點。現有流程有以下技術與行政瓶頸：

1. 【外勤資訊不對等】：
   業務在外勤面對客戶時，常被客戶追問「上次訂的欠貨什麼時候會補寄？到底欠了哪些品項？」。由於業務無法在外直接查閱 VFP ERP，必須聯絡內勤行政查詢，這導致回覆不及時，業務與客端資訊不對等。

2. 【內勤人工整理耗時】：
   內勤行政人員需要定期從 ERP 手動整理、篩選各客戶的欠貨明細，整理成 Excel 發給各業務，效率低且容易產生交期資訊遺漏，且在繁忙的營業時間是一項極大的行政負擔。

3. 【協作備註不同步】：
   欠貨單的「預估到貨日」或「出貨備註」往往是由內勤採購或出貨小姐手動填寫。若將這些備註存在局域網 ERP 中，業務在外勤無法看見；若手動編輯在外部 Excel 中，每次 ERP 導出更新又會覆蓋掉舊備註。

★ 系統設計核心原則：「區網 ERP ➔ 雲端試算表 ➔ Telegram」的雙向資訊共享生態圈。ERP 自動更新，行政手動備註，兩者完美共存。

二、 系統模組與執行流程 (System Modules & Process Flow)
本系統建立了一個「區網 ERP ➔ 雲端試算表 ➔ Telegram」的雙向資訊共享生態圈：

1. 【VFP 欠貨主動掃描引擎】：
   背景常駐程式定期（每 30 分鐘）自動讀取銷貨明細表 (ougo1.dbf)，篩選出「未出貨（out_qty < order_qty）」且未作廢的訂單。
   
2. 【Google Apps Script (GAS) 雲端中繼站】：
   在雲端試算表部署 Webhook 接收 JSON 資料，自動更新試算表。
   
3. 【雲端試算表協作介面】：
   提供給內勤人員隨時編輯「預估到貨日」與「行政備註」的共享平台。
   
4. 【Telegram 快捷查詢模組】：
   在業務 Telegram 鍵盤選單上新增「📋 查詢未出貨」按鈕，快速回報當前同步的欠貨數據。

執行流程簡圖：
   [VFP ERP 資料庫]
          │ (每30分鐘主動掃描)
          ▼
   [本機同步服務 (sync_backorders.js)] ── 傳送 JSON ──► [Google GAS Webhook]
                                                               │
                                                       [比對 Unique Key]
                                                               │
                                                               ▼ (保留人工備註)
                                                    [Google 試算表協作表]
                                                               ▲
                                                               │ (業務一鍵查未出貨)
                                                    [Telegram 查價機器人]

三、 核心技術發想與演算法設計 (Core Technical Details & Algorithms)

1. ☁️ 跨平台雲端主動推送機制 (Cross-Platform Active Push Webhook)
   * 發想由來：
     公司 ERP 位於安全防護的局域網內，外部無法直接進行資料庫連線。
   * 設計實現：
     - 本機 Node.js 背景服務將篩選出的欠貨資料轉為輕量 JSON（包含訂單號 `ORDER_NO`、客戶代號 `CUST_NO`、商品名稱 `PROD_NAME`、欠貨數量 `QTY` 等），透過 HTTP POST 連線主動推送至 Google Apps Script Webhook URL。
     - 這種「內網主動推送，外網唯讀接收」的架構，完全避免了對企業內網防火牆做危險的 Port Forwarding，保證局域網安全。

2. 📝 Google 試算表 Unique-Key 比對防覆蓋演算法 (Unique-Key Merge Algorithm)
   * 發想由來：
     解決每次 ERP 同步時，會覆蓋掉內勤在試算表上「手動填寫的預估到貨日與備註」之痛點。
   * 設計實現：
     - GAS 端接收到新資料後，不會直接暴力覆蓋整張表，而是先將目前的試算表內容讀入記憶體。
     - 系統以「訂單編號 + 商品代碼」（Unique Key）作為唯一識別碼，建立備註的 Map。
     - 流程圖解：
       [GAS 接收新 JSON]
              │
              ▼
       [讀取現有試算表，建立 Map { 'Order123-Prod456': { 到貨日: '5/30', 備註: '已催貨' } }]
              │
              ▼
       [清除工作表，重建最新欠單明細]
              │
              ▼
       [對每一列比對 Unique Key ➔ 從 Map 貼回手動填寫的『預估到貨日』與『備註』]
     - 這實現了「ERP 自動更新，行政手動備註，兩者完美共存」的目標。

3. 📋 Telegram 捷徑一鍵未出貨檢索
   * 發想由來：
     業務隨手即可查詢所有欠貨概況，不需要手動打字。
   * 設計實現：
     - 機器人提供固定選單按鈕。點選後，系統會在 0.2ms 內檢索本機暫存的欠貨清單，並格式化為清晰的排版（列出客戶、品名、欠貨量與最新備註）直接回傳給業務。
     - 【長度截斷防護】：若欠單過多，為防止 Telegram 訊息字數超出限制 (4096字)，系統會自動統計筆數，若超過 20 筆則僅顯示前 20 筆，並提供試算表連結提示使用者查閱完整清單。

四、 安全防護設計 (Security Architecture)
1. 【GAS 安全憑證驗證】：GAS Webhook 設定了自定義的安全 Key。Node.js 發送 POST 時必須在 Header 攜帶此 Key，GAS 驗證相符才處理資料，防止外部惡意寫入 Google 試算表。
2. 【敏感資料過濾】：上傳至 Google 試算表的欄位經過精簡，僅包含出貨所必需的客戶代號、品名、數量，排除其他不相關的敏感進價與利潤等財務資訊。

五、 系統討論重點 (Key Discussion Points)

1. 【同步頻率與歷史資料量流暢度權衡】：
   * **問題**：若累積歷史欠貨紀錄過多，上傳與試算表重組速度會變慢。
   * **分析與建議**：
     - 建議僅同步「未出貨（欠貨）且未結案」的單據。若該單據在 ERP 中已完成補寄或被作廢，則於下次同步時自動從試算表移除。GAS 可設定將已移除的資料封存至另一個「歷史存檔」分頁，以維持主工作表的流暢度。

2. 【內勤編輯與背景同步的衝突防鎖定】：
   * **問題**：當內勤正在 Google 試算表上編輯某個儲存格的瞬間，若背景同步剛好啟動重建工作表，會中斷內勤的輸入。
   * **分析與建議**：
     - 可以採用「編輯狀態檢測」：在 Google Sheet 頂端設立一個「同步中」的提示狀態，或是將同步時間設定在非工作高峰期（如中午休息、傍晚等）。目前每 30 分鐘一次為最優平衡，若內勤反應編輯常被打斷，可調高至每 60 分鐘同步一次。

3. 【業務分權與隱私保護】：
   * **問題**：目前所有授權業務都能看到公司全部欠貨，有隱私洩露風險。
   * **分析與建議**：
     - 未來可在 Telegram 查詢端做分權：當業務 A 點擊按鈕時，Node.js 先讀取該業務的 Telegram ID，並在 ERP 對照表中找出其業務代號（如 Sales01），最後過濾只回傳屬於該業務代號的客戶欠貨清單。
