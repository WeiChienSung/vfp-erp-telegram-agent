一、 設計背景與核心痛點 (Background & Problem Statement)
傳統中小型醫材、化學批發倉庫進行月底盤點時，面臨以下實務痛點：

1. 【手寫抄錄效率低下且極易出錯】：
   倉管人員需要拿著紙本盤點單，逐一對照貨架上的實體數量手寫抄錄，再回到辦公室電腦前，手動登入 ERP 將幾百筆商品數量一筆一筆敲進系統。這不僅耗費雙倍人力，更常因手寫字跡模糊或鍵入疲勞，導致輸入錯誤，帳物不符。

2. 【原廠 ERP 客製與帳號授權昂貴】：
   若要求 ERP 系統商客製手機端行動盤點程式，廠商除了索取數十萬的客製開發費外，通常還要求每台手機都要買額外的使用者授權（Client License），造成沉重的資金負擔。

3. 【倉庫網路訊號不穩與中斷】：
   鋼筋混凝土結構的倉庫深處常有網路死角。若盤點系統必須即時連線資料庫，一旦網路中斷，網頁便會斷線卡死，導致已掃描輸入的資料遺失，引起現場人員極大反彈。

4. 【VFP 併發寫入鎖定衝突】：
   若多人在不同倉庫同時上傳盤點數據，或上傳時核心 ERP 正在進行備份/單據開立，若直接寫入會因獨佔鎖定（Exclusive Lock）失敗而引發系統報錯，甚至損毀 DBF 檔案。

★ 系統設計核心原則：「低成本行動化、離線防丟快取、安全預檢鎖定」。利用輕量化 Node.js 網頁伺服器結合前端 HTML5 掃碼，免授權費，支援百人同時盤點。

二、 系統模組與執行流程 (System Modules & Process Flow)
系統包含以下三個常駐配合的軟硬體模組：

1. 【行動盤點伺服器 (take_server.js)】：
   本機背景服務，開啟 Port 3000 提供網頁服務，並與公網 Localtunnel 串接，動態推播加密網址至 Telegram。
   
2. 【手機掃碼網頁前端 UI (public/index.html & index.js)】：
   採用 HTML5 條碼掃描庫（如 html5-qrcode / quagga），支援直接呼叫手機相機掃碼，並以 HTML5 sessionStorage/localStorage 做離線備援快取。
   
3. 【資料庫寫入腳本 (append_take.ps1)】：
   參數化 PowerShell 腳本，負責在背景執行安全的 DBF 二進位串流附加寫入（Append），向 VFP 的 `take.dbf`（單頭）與 `take1.dbf`（明細）寫入數據。

執行流程簡圖：
    [倉管手機端] ── 掃碼/輸入數量 ──► [本地網頁快取 (防中斷)]
          │ (點擊上傳, 帶安全 Token)
          ▼
    [外網 Localtunnel 轉發]
          │
          ▼
    [take_server.js 背景服務] (Port 3000, 驗證 Token)
          │
          ▼ 呼叫並傳入 -DbDir 及資料
    [append_take.ps1 腳本] (執行預檢鎖, 確認雙檔案未被 ERP 獨佔)
          │
          ▼ 安全附加（Append）
    [Z:\nmedi\take.dbf / take1.dbf 資料庫]

三、 核心技術發想與運作機制 (Core Technical Details & Write Mechanism)

1. 🚀 手機相機條碼掃描與本地斷線快取機制：
   - 【前端相機掃描】：手機前端網頁直接調用 HTML5 Camera API，當偵測到條碼（如 EAN-13、Code-128）後，自動向伺服器發送 API 查出對應商品與品名，並發出觸覺反饋（短震動）。
   - 【離線暫存快取】：若手機在倉庫深處網路中斷，網頁會自動將已掃描的資料存入 `localStorage` / `sessionStorage`。待網路恢復後，點選「重新同步」一鍵批次補傳，保證資料零遺失。
   - 【手機沒電自愈】：若手機中途沒電關機，重啟後網頁會自動從持久化的離線快取中讀取上次未上傳的明細，倉管不需重新掃碼。

2. ⚙️ PowerShell 參數化底層 DBF 數據寫入設計：
   - 【二進位直接寫入】：為避開繁瑣的 VFP 開發工具限制，利用 PowerShell 的 `System.IO.FileStream` 直接讀寫 DBF 檔案的二進位結構。
   - 【虛擬化解耦】：`take_server.js` 與 `append_take.ps1` 完全解耦，資料庫目錄由伺服器動態以 `-DbDir` 參數傳給腳本，免去寫死磁碟機代號的困擾。
   - 【單頭與明細關聯】：自動取得最新盤點單號，向 `take.dbf`（單頭，欄位包含單號、日期、倉管代碼、狀態）寫入一筆，並向 `take1.dbf`（明細，欄位包含單號、項次、條碼、數量、主/次單位、廠商）寫入多筆。

3. 🔒 雙檔案「預檢鎖 (Pre-lock Check)」防衝突控制：
   - 【防檔案毀損機制】：VFP 的單頭與明細表是分開的。若寫入單頭後，明細檔剛好被 ERP 系統備份獨佔鎖定，會造成「有單頭無明細」的懸空壞帳。
   - 【預檢雙鎖】：在寫入前，腳本會先試圖以讀寫模式開啟 `take.dbf` 與 `take1.dbf`。只有當「兩個檔案都可順利開啟且未被獨佔鎖定」時，才會真正執行寫入。若有一方鎖定，則拒絕寫入並回傳忙碌錯誤，通知手機端 5秒後自動重試，保障資料完整性。

四、 Safe Guard & 網路安全防護設計 (Security Architecture)

1. 【外網 API Token 金鑰認證鎖】：
   - 伺服器啟動時，會隨機生成一組 22 位的密碼學安全憑證（API Token），並隨盤點網址推送至 Telegram。
   - 手機端網頁載入後，會自動在後續的 API 請求 Header 中帶上 `x-api-token`。
   - `take_server.js` 後台檢測如果連線來自非本機（外網），則強制要求此金鑰匹配，任何未授權請求一律回傳 401 Unauthorized，徹底封鎖庫存與廠商資料外洩風險。

2. 【本機設定後台保護鎖】：
   - 後台管理路域 `/config` 的 API 路由，會嚴格比對連線來源 IP 必須為 `127.0.0.1` 且不能包含 Localtunnel 轉發標頭。
   - 外網用戶即使拿到 Token，也絕對無法點開 `/config` 修改資料庫路徑或偷看其他 Bot Token，保護安全隱私。

五、 系統討論重點 (Key Discussion Points)

1. 【斷線自癒與快取恢復】：
   - 手機網頁快取是否需要加上定期自動與後台對齊的功能？
   - 離線資料若積壓過多，如何防範瀏覽器 storage 容量限制（通常為 5MB - 10MB）？

2. 【重複掃描與覆蓋衝突】：
   - 倉管對同一個商品不小心掃描了兩次，且輸入不同數量。系統應該要「累加數量」還是「以最後一次為準覆蓋」？目前預設採用累加，但在實務上需與倉庫主管討論，是否提供「覆蓋」的選項。

3. 【系統商維護與鎖定安全】：
   - 當 ERP 系統商進行整庫維護或重組索引時，會對所有 DBF 加上長達數小時的獨佔鎖。
   - 影子系統在遇到此狀況時，除了前台提示「系統維護中」外，是否需要將盤點資料長效快取在伺服器端，等維護結束後再自動補寫入？
