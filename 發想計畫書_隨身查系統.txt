一、 設計背景與核心痛點 (Background & Problem Statement)
傳統中小型醫材、化工批發企業使用以 Visual FoxPro (VFP) 為核心的舊型 ERP 系統。
在日常的外勤報價、倉儲盤點中，面臨以下三個技術瓶頸：

1. 【外勤資訊孤島】：
   業務在外勤面對客戶時，無法即時查詢即時庫存與售價。例如客戶詢問「某防護面罩一箱多少？有現貨嗎？」，業務必須致電回辦公室請內勤在 ERP 中查詢。這除了耗費內勤行政人力、增加電話溝通成本外，若遇上內勤電話佔線，常需等待 10-20 分鐘才能答覆，極易導致客戶流失。
   
2. 【成交報價錯誤】：
   醫療器材與化工原料商品規格繁多（例如針頭有 21G、23G、25G 等不同長度與粗細），且不同客戶有其專屬的成交單價。業務在外勤若缺乏歷史成交價做參考，常因口頭憑印象報價錯誤，造成公司利潤損失或客戶感到不公而損害商譽。

3. 【資料庫獨佔鎖定衝突 (Exclusive Lock)】：
   核心 ERP 採用本機/網路共用磁碟（Z 槽 NAS）的 DBF 檔案儲存。若外部程式直接寫入或進行全表檢索，會觸發 VFP 的 Exclusive Lock，導致正在使用核心 ERP 的內勤人員電腦當機、單據無法儲存，嚴重時甚至造成資料毀損。

★ 系統設計核心原則：「零侵入、唯讀安全、極速響應」。藉由本機 Node.js 背景服務作為緩衝代理，將資料無損地打通至 Telegram 機器人端。

二、 系統模組與執行流程 (System Modules & Process Flow)
系統由三大背景常駐常開模組組成，全部由 Windows 的啟動資料夾 (Startup) 透過 VBS 隱形代理程式在開機時默默喚醒：

1. 【Telegram 查價機器人 (telegram_bot.js)】：
   負責 24 小時輪詢處理使用者輸入，支援多 Slots（多個機器人分流處理查價、盤點、歷史售價等）。
   
2. 【本機設定伺服器 (take_server.js)】：
   提供本地端可視化 GUI 後台介面 (http://localhost:3000/config) 管理機器人權限與功能開關。
   
3. 【本機 VFP 唯讀讀取引擎】：
   負責對 NAS Z 槽之 DBF 檔案進行無損、極速的二進位檢索。

執行流程簡圖：
   [業務 Telegram 手機端]
          │ (輸入關鍵字, 如 "3m 1.5 膚")
          ▼
   [Telegram 伺服器] ── 輪詢 (Long Polling) ──► [telegram_bot.js 背景服務]
                                                       │
                                            ┌──────────┴──────────┐
                                            ▼                     ▼
                                    [VFP 商品庫存檔]       [1.3GB 銷貨歷史大檔]
                                      (ougo.dbf)            (ougo1.dbf)
                                   (耗時 < 10ms)       (逆向分塊檢索, < 200ms)

三、 核心技術發想與演算法設計 (Core Technical Details & Algorithms)

1. 🚀 1.3GB 銷貨大檔逆向分塊 Buffer 檢索演算法 (Reverse Block-Buffer Reading)
   * 發想由來：
     銷貨明細表 (ougo1.dbf) 紀錄了歷史上百萬筆的成交資料，檔案大小達 1.3 GB。傳統搜尋套件需載入整張表或全表掃描，在慢速網域下需耗時數分鐘，且會導致伺服器 CPU 瞬間衝高 100%。
   * 演算法設計：
     - 本系統不採用全表載入，而是透過 Node.js `fs.readSync` 實作二進位「逆向分塊讀取」。
     - 自檔案尾端（最新交易時間）逆向以 `CHUNK_RECORDS = 5000` 筆為一塊（約為最近 1 週的銷貨量）讀入記憶體 Buffer。
     - 【記憶體防護 (GC Protection)】：為了防止 JavaScript 閉包一直引用大 Buffer 造成垃圾回收 (GC) 失敗引發 Memory Leak，系統在匹配成功時，立即將其屬性拷貝實體化 (Materialize) 為一個新的輕量物件，然後丟棄原 Buffer。
     - 【惰性解碼 (Lazy Decoding)】：因為 Big5 中文字元集解碼（使用 TextDecoder）非常消耗 CPU，我們為每一筆 record 實作了 `Object.defineProperty` 的 Getter 屬性。只有當該列符合「特定客戶 ID」需要顯示給業務看時，才會真正進行中文字元解碼，並將解碼結果快取。此優化將大檔檢索時間從數分鐘壓縮至 **200 毫秒以內**！

2. 🔍 繁簡中文同音異體字自動相容機制 (Character Mapping Dictionary)
   * 發想由來：
     建檔人員輸入的品名常有錯別字或個人習慣（例如：溼疹與濕疹、1寸與1吋），導致業務輸入正確字彙反而查無商品。
   * 設計實現：
     - 系統內建「異體同音字雙向標準化映射表」：
       - `溼` ↔ `濕`、`寸` ↔ `吋`、`尺` ↔ `呎`、`台` ↔ `臺`
       - 繁簡對應：`國` ↔ `国`、`醫` ↔ `医`、`藥` ↔ `药`、`針` ↔ `针` 等。
     - 所有搜尋詞在進入比對引擎前，會統一被標準化（Normalized），然後再與標準化後的商品品名比對，完全杜絕因錯別字查無資料的痛點。
     - 流程：`使用者輸入: "3m1.5吋濕"` ➔ `標準化: "3m1.5寸溼"` ➔ `與系統內標準化品名進行比對` ➔ `匹配成功`

3. 🔍 中英數混合免空格智慧分詞與權重打分 (Smart Tokenization & Ranking)
   * 發想由來：
     使用者在手機輸入時極不方便鍵入空白鍵，常打出「3m1吋膚」或「針頭23g」。
   * 設計實現：
     - 分詞引擎使用正規表示式，自動在【中文】、【英文字母】與【數字（支援 2.5 等浮點數）】的邊界進行智慧切分。
     - 例如將「3m1.5吋膚」自動切分為三個 Token：`3m`、`1.5`、`吋`、`膚`。
     - 所有 Token 採交集 (AND) 搜尋比對。
     - **打分排序機制 (Ranking)**：
       - 完全連續匹配（沒有被切分或順序完全正確）給予最高分 (100分)。
       - 分詞匹配（包含所有 Token 但位置分散）給予基礎分 (10分)。
       - 結果以得分降序排序，確保最精確的規格商品永遠排在搜尋結果的第一位。

四、 安全防護設計 (Security Architecture)
1. 【本機環回防護鎖】：設定後台 (`take_server.js`) 嚴格限制僅監聽 `127.0.0.1:3000` (localhost)，拒絕來自區域網路或外部網域的任何設定讀寫請求，確保 Bot Token 與敏感憑證不外洩。
2. 【對話權限動態鎖】：Telegram 機器人設定了 `authorizedChats` 授權清單，僅允許清單內的業務對話 ID 查價，並禁止外部人員存取。
3. 【防 DoS 查詢防禦】：為防範惡意人員在 Telegram 發送超長垃圾文字轟炸伺服器，機器人輸入字數限制在 200 字以內，且單次搜尋關鍵字上限為 5 個。

五、 系統討論重點 (Key Discussion Points)

1. 【智慧分詞之匹配深度與精準度平衡】：
   * **問題**：目前採用交集 (AND) 搜尋，當使用者輸入過多關鍵字（如：3m 1.5 膚 綠）時，只要有一個 Token 沒對上就查無資料。
   * **優選方案**：
     - 方案 A (維持現狀)：維持交集 AND 搜尋。優點是結果極其精準，缺點是容錯率低。
     - 方案 B (權重模糊化)：改為聯集 OR 搜尋，並根據匹配的 Token 數量給予不同權重得分。優點是容錯率高，缺點是若資料庫大，可能會查出很多無關的雜訊商品。
   * **建議**：目前維持方案 A，若未來業務反應查無資料率過高，再引進方案 B 的部分匹配機制。

2. 【銷貨大檔逆向檢索之搜尋邊界】：
   * **問題**：若遇到極冷門、甚至歷史上無交易記錄的商品，程式會一路逆向檢索整個 1.3GB 檔案至最前端，耗費較多本機 I/O。
   * **優選方案**：
     - 方案 A (無限制掃描)：不管如何都掃描到檔頭。優點是絕對不會遺漏歷史成交，缺點是極冷門品項搜尋時間會拉長。
     - 方案 B (設定搜尋上限)：當搜尋回溯超過 100,000 筆或時間超過 6 個月，即停止搜尋並提示「近期無交易紀錄」。
   * **建議**：建議採用方案 B，將搜尋邊界限制在「近 1 年內或 20 萬筆交易紀錄」，以確保系統在極端情況下仍能維持在 200ms 內響應。

3. 【行動端排版欄位精簡化】：
   * **問題**：Telegram 螢幕寬度有限。如果在查價結果中回傳過多欄位（品名、規格、原廠程式碼、現有庫存、預售價、歷史成交價等），會導致訊息洗版。
   * **建議**：精簡回傳格式，以「粗體顯示品名」，緊接著顯示「庫存」與「預售單價」，並提供按鈕供需要時再回查歷史價，避免單次訊息字數過多。
