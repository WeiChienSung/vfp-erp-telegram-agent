========================================================================
   【VFP ERP 價格與庫存隨身查及智慧推播系統 - 深度發想與系統架構設計書】
========================================================================

一、 設計背景與核心痛點 (Background & Problem Statement)
傳統中小型醫材、化工批發企業使用以 Visual FoxPro (VFP) 為核心的舊型 ERP 系統。
在日常的外勤報價、倉儲盤點中，面臨以下三個技術瓶頸：
1. 【外勤資訊孤島】：業務在客端現場無法即時查詢即時庫存與售價，需打電話回內勤，耗費人力且溝通成本高。
2. 【成交報價錯誤】：商品規格繁多、且不同客戶有其專屬的成交單價。業務若缺乏歷史成交價參考，常因口頭報價錯誤造成公司利潤損失或商譽受損。
3. 【資料庫獨佔鎖定衝突】：核心 ERP 採用本機/網路共用磁碟（Z 槽 NAS）的 DBF 檔案儲存。若外部程式直接寫入或進行全表檢索，會觸發 VFP 的 Exclusive Lock，導致核心 ERP 當機或資料毀損。

本系統本著「零侵入、唯讀安全、極速響應」的架構原則，藉由本機 Node.js 背景服務作為緩衝代理，將資料無損地打通至 Telegram 機器人端。

二、 系統模組與執行流程 (System Modules & Process Flow)
系統由三大背景常駐常開模組組成，全部由 Windows 的啟動資料夾 (Startup) 透過 VBS 隱形代理程式在開機時默默喚醒：
1. 【Telegram 查價機器人 (telegram_bot.js)】：負責 24 小時輪詢處理使用者輸入，支援多 Slots（多個機器人分流處理查價、盤點、歷史售價、推播）。
2. 【本機設定伺服器 (take_server.js)】：提供本地端可視化 GUI 後台介面 (http://localhost:3000/config) 管理機器人權限與推播。
3. 【智慧推播調度器】：依設定時間自動觸發庫存快照比對，發送斷貨警報。

三、 核心技術發想與演算法設計 (Core Technical Details & Algorithms)

1. 🚀 1.3GB 銷貨大檔逆向分塊 Buffer 檢索演算法 (Reverse Block-Buffer Reading)
   * 發想由來：銷貨明細表 (ougo1.dbf) 紀錄了歷史上百萬筆的成交資料，檔案大小達 1.3 GB。傳統搜尋套件需載入整張表或全表掃描，在慢速網域下需耗時數分鐘，且會導致伺服器 CPU 瞬間衝高 100%。
   * 演算法設計：
     - 本系統不採用全表載入，而是透過 Node.js `fs.readSync` 實作二進位「逆向分塊讀取」。
     - 自檔案尾端（最新交易時間）逆向以 `CHUNK_RECORDS = 5000` 筆為一塊（約為最近 1 週的銷貨量）讀入記憶體 Buffer。
     - 【記憶體防護 (GC Protection)】：為了防止 JavaScript 閉包一直引用大 Buffer 造成垃圾回收 (GC) 失敗引發 Memory Leak，系統在匹配成功時，立即將其屬性拷貝實體化 (Materialize) 為一個新的輕量物件，然後丟棄原 Buffer。
     - 【惰性解碼 (Lazy Decoding)】：因為 Big5 中文字元集解碼（使用 TextDecoder）非常消耗 CPU，我們為每一筆 record 實作了 `Object.defineProperty` 的 Getter 屬性。只有當該列符合「特定客戶 ID」需要顯示給業務看時，才會真正進行中文字元解碼，並將解碼結果快取。此優化將大檔檢索時間從數分鐘壓縮至 **200 毫秒以內**！

2. 🔍 繁簡中文同音異體字自動相容機制 (Character Mapping Dictionary)
   * 發想由來：建檔人員輸入的品名常有錯別字或個人習慣（例如：溼疹與濕疹、1寸與1吋），導致業務輸入正確字彙反而查無商品。
   * 設計實現：
     - 系統內建「異體同音字雙向標準化映射表」：
       - `溼` ↔ `濕`
       - `寸` ↔ `吋`
       - `尺` ↔ `呎`
       - `台` ↔ `臺`
       - 繁簡對應：`國` ↔ `国`、`醫` ↔ `医`、`藥` ↔ `药`、`針` ↔ `针` 等。
     - 所有搜尋詞在進入比對引擎前，會統一被標準化（Normalized），然後再與標準化後的商品品名比對，完全杜絕因錯別字查無資料的痛點。

3. 🔍 中英數混合免空格智慧分詞與權重打分 (Smart Tokenization & Ranking)
   * 發想由來：使用者在手機輸入時極不方便鍵入空白鍵，常打出「3m1吋膚」或「針頭23g」。
   * 設計實現：
     - 分詞引擎使用正規表示式，自動在【中文】、【英文字母】與【數字（支援 2.5 等浮點數）】的邊界進行智慧切分。
     - 例如將「3m1.5吋膚」自動切分為三個 Token：`3m`、`1.5`、`吋`、`膚`。
     - 所有 Token 採交集 (AND) 搜尋比對。
     - **打分排序機制 (Ranking)**：完全連續匹配給予最高分 (100分)，分詞匹配給予基礎分 (10分)，結果以得分降序排序，確保最精確的規格商品永遠排在搜尋結果的第一位。

4. 🔔 智慧「斷貨瞬時警報」防疲勞推送機制 (Smart Push)
   * 發想由來：一般系統的低庫存通知會在商品庫存為 0 時天天發送，造成業務每天收到幾百條重複的斷貨訊息，最後人員直接無視警報。
   * 設計實現：
     - 系統在每日定時推播時，會生成一份當下的庫存狀態快照 (`stock_snapshot.json`)。
     - 比對快照中，**只有當某個商品的庫存狀態「從上次大於 0 的狀態，在今天剛好變為 0」時，才會觸發瞬時斷貨警報**。
     - 如果某商品已經斷貨很久（上次就是 0，這次也是 0），或是已經停售零庫存，系統會自動過濾。這大幅降低了 90% 的垃圾警報，讓採購人員只專注於「剛剛賣斷貨」的品項。

5. 🔔 1分鐘模組化即時售罄告警與一鍵歷史成交查詢 (Modular Instant Alert & Callback Query)
   * 發想由來：業務與採購常需在熱銷商品「庫存歸零的瞬間」獲得通知以立即啟動補貨機制，同時在收到斷貨通知時，最急迫的動作往往是了解「該商品最近賣給誰、售價多少、是否還有未出貨單據」以進行調度。如果每一次都要在手機上手動輸入關鍵字查歷史價或未出貨單，操作極其繁瑣。
   * 設計實現：
     - 【即時售罄過濾器 (zero_stock_alert.js)】：以獨立模組常駐，每分鐘輕量比對 VFP 庫存檔。透過記憶體狀態機追蹤變量（`prevStock > 0 && currentStock <= 0`），只有在商品庫存「真正歸零的一瞬間」才發送告警，並實作冷啟動保護（Cold-Start Guard），避免重啟服務時因狀態重置而重發告警。
     - 【防抖動批次發送 (Batching Alert)】：若同一分鐘內有多個商品同時歸零，系統會將告警訊息自動打包合併為單一推播，防止多次呼叫 Telegram API 導致被官方限流（Rate Limit）。
     - 【Telegram 行動 Inline Keyboard 按鈕 (`hist:MNO`)】：告警推播中直接嵌入 Inline 查詢按鈕。使用者點擊後，Telegram 會觸發 Callback Query，後台接到請求後，以 200ms 的極速讀取該商品歷史成交明細並以快閃訊息回覆，搭配 `answerCallbackQuery` 消除 UI 轉圈等待，大幅縮短外勤與內勤的反應時間。

四、 安全防護設計 (Security Architecture)
1. 【本機環回防護鎖】：設定後台 (`take_server.js`) 嚴格限制僅監聽 `127.0.0.1:3000` (localhost)，拒絕來自區域網路或外部網域的任何設定讀寫請求，確保 Bot Token 與敏感憑證不外洩。
2. 【對話權限動態鎖】：Telegram 機器人設定了 `authorizedChats` 授權清單，僅允許清單內的業務對話 ID 查價，並禁止外部人員存取。
3. 【防 DoS 查詢防禦】：為防範惡意人員在 Telegram 發送超長垃圾文字轟炸伺服器，機器人輸入字數限制在 200 字以內，且單次搜尋關鍵字上限為 5 個。
