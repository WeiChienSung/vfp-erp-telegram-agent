一、 系統基本架構與開機啟動鏈
========================================================================
1. 【程式路徑】：
   * 本機實體檔案存放於：`C:\agy_Add_on\ERP 隨身查與盤點\`。
   * 內含核心檔案：`telegram_bot.js` (機器人主程式)、`config.json` (設定檔)。
2. 【安全常駐啟動鏈】：
   為了防範 Windows Session 0 執行權限限制與解決 WScript 的中文路徑亂碼 Bug，採用了「純英文 VBS + 中文相對路徑 BAT」的混合啟動方式：
   * 開機時，系統會自動在背景隱形拉起 `C:\agy_Add_on\run_query.vbs`（無任何黑視窗）。
   * VBS 會呼叫 `C:\agy_Add_on\run_query.bat`，在批次檔中以 CP950(Big5) 編碼執行相對路徑切換（`cd /d "%~dp0\ERP 隨身查與盤點"`）並以 Node.js 拉起 `telegram_bot.js`。
   * 日誌輸出採追加模式（`>> telegram_bot.log 2>&1`），保證夜間崩潰時能完整保留錯誤日誌。


二、 功能特徵 1：商品一般查價（stock.dbf 搜尋排序引擎）
========================================================================
1. 【中英數混合分詞優化】：
   當同仁輸入中英數混合詞（如 `3m大白` 或 `3M大白`）時，系統使用優化後的正則分詞器（`/[a-z0-9]+(?:\.[0-9]+)?|[\u4e00-\u9fa5]+/gi`，中文字段加上 `+` 量詞）。
   * 效果：`3M大白` 會被拆分為精確的 `["3m", "大白"]`，而不會碎裂為 `["3m", "大", "白"]`，保證了「無空格」與「有空格」的查詢結果 100% 達到行為一致，排除無關雜訊。
2. **【四階梯相關性打分機制（Relevance Scoring）】**：
   為了讓最相關的商品浮動在 Telegram 列表最頂端，系統不使用二元篩選，而是採用智慧排序：
   * **Tier 1: 緊湊完全匹配（100 分）** ➔ 輸入的完整字串（去空格）完全出現在品名、代號或條碼中。
   * **Tier 2: 順序連續匹配（70~85 分）** ➔ 關鍵字在品名中按順序出現，並依據字元間距給予「緊湊度獎勵分」（中間夾雜的字元越少，分數越高）。
   * **Tier 3: 無序 AND 匹配（15~25 分）** ➔ 關鍵字全部符合但順序相反（如輸入 `大白 3M`，匹配 `3M...大白`）。
   * **短品名密度獎勵（最大 +10 分）** ➔ 品名越短，關鍵字密度越高，額外加分，確保精準規格品排在最前面。
3. 【停售商品警示】：
   如果商品名稱包含「停產」、「停用」或「停售」，機器人在回傳列表時會在其名稱旁強烈標記：`⛔ [已停售]`，警告業務切勿向藥局報價。
4. 【多商品關鍵字拆分查價降級】：
   當業務一口氣查詢多個商品（如輸入：`克淋溼 3m 大白`）且完全匹配結果為 0 時，系統自動將其拆開，在記憶體中平行檢索，並分組回傳（如：與「克淋溼」相關的商品、與「大白」相關的商品），避免回傳「查無此商品」的尷尬。
5. 【記憶體快取與增量更新（9ms 極速）】：
   系統在記憶體中建立 `stockCache`，僅在 `stock.DBF` 的修改時間變更時才重新讀檔解析。查詢耗時由 **250ms 暴降至 9ms（提升 25 倍）**，徹底消除 UNC 網路共享磁碟的 I/O 瓶頸。


三、 功能特徵 2：客戶歷史售價查詢（ougo1.dbf 檢索）
========================================================================
1. 【輸入格式】：
   格式為：`[客戶名稱/代號] [商品關鍵字]`（例如：`大安 3m` 或是 `大安 3m大白`）。
2. 【客戶模糊分類器】：
   系統首先讀取 `CUST.DBF`（亦建立了記憶體快取以求 0ms 響應），將「大安」模糊匹配至官方的 `[G008] 大安` 客戶。
3. 【超大檔案 Chunk 逆向掃描】：
   銷貨明細檔 `ougo1.dbf` 高達 1.3 GB。系統採用**逆向分塊掃描（queryDbfOptimized，從檔案尾端往前讀取）**，只撈取該客戶最近發生的 3 次成交紀錄（包含：成交日期、商品品名、數量、單位、單價、小計金額）。
4. 【歷史成交中的停售標記】：
   若歷史成交明細中的商品在目前基本庫中已被標記為停售，會在歷史成交紀錄的日期旁同步標記 `⛔ [已停售]`，與一般查價對齊。
5. 【商品預檢效能快速攔截】：
   在開啟並掃描 1.3 GB 的 `ougo1.dbf` 之前，系統會先確認商品關鍵字是否存在於商品基本庫中。若基本庫中根本查無此商品，則**直接在記憶體中 Early Return 快速攔截**，完全避免無謂的 1.3 GB 磁碟讀取。


四、 五大安全與強健性防禦設計
========================================================================
1. 【Telegram 授權白名單安全守衛】：
   除了 `/start` 與 `/register` 註冊指令外，任何未列在 `config.json` 內 `authorizedChats` 白名單中的 Chat ID（陌生人），一律進行物理攔截，機器人拒絕處理任何查詢，防止陌生人竊取商業資料。
2. 【CPU 忙等待忙碌鎖死防範（sleepSync）】：
   當 ERP 正在進行備份或獨佔鎖定 DBF 檔案時，系統如果用無窮迴圈（busy-waiting）等待會將 Node.js 的單執行緒卡死。
   系統引入基於 `SharedArrayBuffer` 與 `Atomics.wait` 的極致同步睡眠 `sleepSync(ms)`，在重試等待的 150ms 期間**主動向操作系統讓出 CPU 時間**，消滅 CPU 熱耗竭隱憂。
3. 【網路磁碟瞬斷之自癒重試】：
   將 `fs.existsSync` 和 `fs.statSync` 等檔案系統操作，一併納入 3 次自癒重試迴圈內。當 UNC 網路共享路徑發生瞬時抖動或中斷時，會進行 150ms 間隔的重試，避免服務直接崩潰。
4. 【查價防刷限制（Rate Limiter）】：
   為防範業務高頻點擊或惡意灌水造成資料庫 I/O 過載，每個 `chatId` 的查詢冷卻時間限制為 1.5 秒。低於此限制會回覆：`⚠️ 您的查詢頻率過快，請稍候再試。`
5. 【防範 TypeError 導致啟動崩潰】：
   在載入設定與輪詢程式中，對 `features`、`authorizedChats` 陣列加入了安全防護（`|| []`）與 `.catch(...)` 未捕獲 Promise Rejection 防護，確保機器人服務 24 小時不間斷運行。


五、 移除的過時功能（盤點與外網穿透之淨化說明）
========================================================================
為了響應「100% 本地化且無任何外網安全/穿透維運隱憂」的安全原則，系統已徹底進行了自癒淨化：
1. 物理刪除了 `take_server.js` (原 Web 盤點與設定後台)、`append_take.ps1` (寫入 VFP 盤點資料庫腳本)、`active_url.txt` (穿透網址檔)。
2. 物理刪除了整個 `public/` 前端網頁目錄（包括原盤點主頁與設定網頁）。
3. 移除了 `localtunnel` 外部網路穿透模組，背景絕無任何對外穿透連接與存活檢查執行緒。
4. 移除了啟動與重連時的 Telegram 自動群組推播，保持完全靜默。
5. 目前背景常駐進程**僅剩下 1 個 node.exe**（對應 `telegram_bot.js` 查價機器人），本機 Port 3000 已無任何 Listen 監聽，完全消除了所有來自外網的存取管道。
